# MEG Data Model

Derived from `MEG_MASTER_PRD_v4.1_patched.md` v4.1. This document specifies the Phase 0A shared-rail data contracts and the adjacent research-lake views needed to keep runtime and research data planes consistent.

## Scope

This file documents:

- Canonical identifiers used across events, Redis payloads, Postgres tables, and DuckDB views.
- Versioned event payloads for the shared rail.
- Redis channel ownership and payload contracts.
- Phase 0A Postgres operational-journal DDL.
- DuckDB normalized research views expected by Phase 0B and later replay work.
- Migration notes from legacy `market_id` usage.

It does not define strategy implementation, weather strategy code, whale strategy code, or autonomous execution logic.

## Canonical identifiers

All shared-rail contracts use this identifier triple:

| Field | Type | Required | Meaning | Routing allowed? |
|---|---:|---:|---|---:|
| `condition_id` | text | yes | Polymarket market condition identifier, represented as a hex string. | yes |
| `token_id` | text | yes | Polymarket outcome token identifier, represented as a numeric string. | yes |
| `outcome` | text enum | yes | Outcome label derived from token metadata: `YES` or `NO`. | yes |
| `market_slug` | text | no | Human-readable display slug. | no |

Rules:

1. `market_id` is deprecated for Phase 0A and must not appear in new schemas, Redis payloads, Postgres tables, or DuckDB normalized views.
2. `market_slug` is display-only and must not be used for matching, routing, or order placement.
3. `token_id` is the venue-native key for order books and order placement.
4. Strategy-facing order intent may use `(condition_id, outcome, side, notional, price_constraint)`, but the execution rail must resolve that intent to `token_id` before venue interaction.

## Common scalar conventions

| Convention | Type | Notes |
|---|---:|---|
| IDs generated by MEG | `uuid` | `signal_id`, `proposal_id`, `execution_id`, `lot_id`. |
| Timestamps in events | integer milliseconds | Field names use `_ms`, e.g. `ts_emitted_ms`. |
| Timestamps in Postgres | `timestamptz` | Preserve UTC. |
| Prices/probabilities | numeric | Store as decimals in `[0, 1]` unless explicitly bps or pUSD. |
| Money | numeric | pUSD amounts use `_pusd` suffix. |
| Shares | numeric | Outcome-token share quantity. |
| JSON metadata | `jsonb` | Store score breakdowns, raw payload references, and rail diagnostics. |
| Mode | text enum | `replay`, `paper`, `live`. |
| Side | text enum | `BUY`, `SELL`. |

## Event schemas

Every event has a `schema_version` integer. Phase 0A starts at `1`.

### `RawWhaleFill`

Authoritative decoded fill event from the Polygon receipt decoder. It is documented here because Phase 0A owns the shared identifier and bus contract, even though the producer is repaired in Phase 0C.

```json
{
  "schema_version": 1,
  "event_type": "raw_whale_fill",
  "venue": "polymarket",
  "ts_event_ms": 1762819205123,
  "block_number": 73123456,
  "tx_hash": "0x...",
  "condition_id": "0x...",
  "token_id": "12345678901234567890",
  "market_slug": "will-btc-be-above-120k-on-june-30",
  "outcome": "YES",
  "wallet_address": "0xwallet",
  "maker_address": "0xmaker",
  "aggressor_side": "BUY",
  "price": 0.63,
  "size_shares": 720.0,
  "notional_pusd": 453.60,
  "fee_pusd": 0.18,
  "neg_risk": false,
  "source": "polygon_receipt+user_ws"
}
```

Required validation:

- `event_type = raw_whale_fill`.
- `condition_id`, `token_id`, and `outcome` are present.
- `outcome in ('YES', 'NO')`.
- `aggressor_side in ('BUY', 'SELL')`.
- `price between 0 and 1`.
- `size_shares >= 0`, `notional_pusd >= 0`, `fee_pusd >= 0`.

### `Signal`

Output from any strategy signal worker. Strategies emit signals; the decision agent consumes them and may create a `TradeProposal`.

```json
{
  "schema_version": 1,
  "event_type": "signal",
  "signal_id": "uuid",
  "strategy_id": "whale_lead_lag",
  "ts_emitted_ms": 1762819210000,
  "ts_expires_ms": 1762819510000,
  "condition_id": "0x...",
  "token_id": "12345678901234567890",
  "market_slug": "will-btc-be-above-120k-on-june-30",
  "outcome": "YES",
  "side": "BUY",
  "model_probability": 0.71,
  "market_probability": 0.63,
  "edge": 0.08,
  "score_breakdown": {},
  "half_life_seconds": 240,
  "metadata": {}
}
```

Required validation:

- `ts_expires_ms > ts_emitted_ms`.
- `model_probability`, `market_probability`, and `edge` are finite numeric values.
- `side in ('BUY', 'SELL')`.
- The signal does not bypass the decision agent or approval queue.

## Redis contracts

| Channel/key | Producer | Consumer | Payload | Phase | Notes |
|---|---|---|---|---|---|
| `raw_whale_trades` | Polygon fill decoder | Pre-filter pipeline | `RawWhaleFill` | 0C producer, 0A contract | Shared contract only in 0A. |
| `qualified_whale_trades` | Pre-filter pipeline | Whale signal worker | `QualifiedWhaleFill` | 0C producer/consumer, 0A contract | Shared contract only in 0A. |
| `signal_events` | Strategy signal workers | Decision agent | `Signal` | 0A | Strategy-agnostic channel. |
| `trade_proposals` | Decision agent | Telegram bot, dashboard | `TradeProposal` | 0A | Single proposal bus for all strategies. |
| `execution_requests` | Telegram approval handler | Execution rail | `ExecutionRequest` | 0A | Only approved, non-expired proposals. |
| `fills.user` | Polymarket user WebSocket | Reconciler, journal writer | `UserFill` | 0A | Authoritative fill confirmation. |
| `bot_alerts` | Any module | Telegram bot | Alert JSON | 0A | Used for heartbeats, stale cache, reconcile alerts. |
| `market:{token_id}:state` | CLOB market-state feed | Pre-filter, execution, monitoring | `MarketState` | 0A | Concrete member of the master PRD's `market:{token_id}:*` namespace; latest token state must include timestamp. |
| `market:{token_id}:book` | CLOB market-state feed | Paper execution simulator | Order book snapshot JSON | 0A | Concrete member of the master PRD's `market:{token_id}:*` namespace; optional if simulator needs book depth. |
| `replay.clock` | Replay harness | All test-mode services | Deterministic clock tick | Later/replay | Documented now for contract stability. |

Ownership rule: no module writes to a channel it does not own in this table.

## Postgres operational schema

The operational journal records what the system did. It must not store raw market-data history that belongs in DuckDB/Parquet.

### `signal_journal`

```sql
CREATE TABLE signal_journal (
    id BIGSERIAL PRIMARY KEY,
    signal_id UUID NOT NULL UNIQUE,
    schema_version INTEGER NOT NULL DEFAULT 1,
    strategy_id TEXT NOT NULL,
    condition_id TEXT NOT NULL,
    token_id TEXT NOT NULL,
    outcome TEXT NOT NULL CHECK (outcome IN ('YES', 'NO')),
    market_slug TEXT,
    side TEXT NOT NULL CHECK (side IN ('BUY', 'SELL')),
    model_probability NUMERIC(10, 6),
    market_probability NUMERIC(10, 6),
    edge NUMERIC(10, 6),
    score_breakdown JSONB NOT NULL DEFAULT '{}'::jsonb,
    half_life_seconds INTEGER,
    ts_emitted TIMESTAMPTZ NOT NULL,
    ts_expires TIMESTAMPTZ NOT NULL,
    proposal_id UUID,
    metadata JSONB NOT NULL DEFAULT '{}'::jsonb,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    CHECK (ts_expires > ts_emitted)
);

CREATE INDEX idx_signal_journal_signal_id ON signal_journal (signal_id);
CREATE INDEX idx_signal_journal_strategy_id ON signal_journal (strategy_id);
CREATE INDEX idx_signal_journal_condition_id ON signal_journal (condition_id);
CREATE INDEX idx_signal_journal_ts_emitted ON signal_journal (ts_emitted);
```

### `proposal_current_state` and `proposal_state_transitions`

Proposal audit history is transition-safe and immutable. `proposal_current_state` stores the latest state for fast lookup; `proposal_state_transitions` is append-only and records every create, approve, reject, defer, and expire transition.

```sql
CREATE TABLE proposal_current_state (
    id BIGSERIAL PRIMARY KEY,
    proposal_id UUID NOT NULL UNIQUE,
    schema_version INTEGER NOT NULL DEFAULT 1,
    signal_id UUID NOT NULL,
    strategy_id TEXT NOT NULL,
    current_state TEXT NOT NULL CHECK (current_state IN ('created', 'approved', 'rejected', 'deferred', 'expired')),
    condition_id TEXT NOT NULL,
    token_id TEXT NOT NULL,
    outcome TEXT NOT NULL CHECK (outcome IN ('YES', 'NO')),
    market_slug TEXT,
    side TEXT NOT NULL CHECK (side IN ('BUY', 'SELL')),
    proposed_notional_pusd NUMERIC(18, 6) NOT NULL,
    approved_notional_pusd NUMERIC(18, 6),
    price_constraint NUMERIC(10, 6) NOT NULL,
    model_probability NUMERIC(10, 6),
    market_probability NUMERIC(10, 6),
    estimated_edge NUMERIC(10, 6),
    risk_envelope JSONB NOT NULL DEFAULT '{}'::jsonb,
    summary TEXT NOT NULL,
    created_at TIMESTAMPTZ NOT NULL,
    expires_at TIMESTAMPTZ NOT NULL,
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    metadata JSONB NOT NULL DEFAULT '{}'::jsonb,
    CHECK (expires_at > created_at)
);

CREATE TABLE proposal_state_transitions (
    id BIGSERIAL PRIMARY KEY,
    proposal_id UUID NOT NULL REFERENCES proposal_current_state (proposal_id),
    previous_state TEXT CHECK (previous_state IN ('created', 'approved', 'rejected', 'deferred', 'expired')),
    next_state TEXT NOT NULL CHECK (next_state IN ('created', 'approved', 'rejected', 'deferred', 'expired')),
    transitioned_at TIMESTAMPTZ NOT NULL,
    actor_id TEXT,
    reason TEXT,
    notes TEXT,
    approval_latency_ms INTEGER,
    metadata JSONB NOT NULL DEFAULT '{}'::jsonb
);

CREATE INDEX idx_proposal_current_state_proposal_id ON proposal_current_state (proposal_id);
CREATE INDEX idx_proposal_current_state_signal_id ON proposal_current_state (signal_id);
CREATE INDEX idx_proposal_current_state_current_state ON proposal_current_state (current_state);
CREATE INDEX idx_proposal_current_state_created_at ON proposal_current_state (created_at);
CREATE INDEX idx_proposal_state_transitions_proposal_id ON proposal_state_transitions (proposal_id);
CREATE INDEX idx_proposal_state_transitions_transitioned_at ON proposal_state_transitions (transitioned_at);
```

### `trade_journal`

Central execution and reconciliation table. `partially_filled` means an in-progress partial fill; `partially_filled_closed_out` is the explicit terminal state for a partial fill that has been closed out.

```sql
CREATE TABLE trade_journal (
    id BIGSERIAL PRIMARY KEY,
    execution_id UUID NOT NULL UNIQUE,
    proposal_id UUID NOT NULL,
    signal_id UUID,
    schema_version INTEGER NOT NULL DEFAULT 1,
    strategy_id TEXT NOT NULL,
    mode TEXT NOT NULL CHECK (mode IN ('replay', 'paper', 'live')),
    state TEXT NOT NULL CHECK (state IN ('requested', 'submitted', 'posted', 'filled', 'partially_filled', 'partially_filled_closed_out', 'cancelled', 'rejected', 'reconciled')),
    condition_id TEXT NOT NULL,
    token_id TEXT NOT NULL,
    outcome TEXT NOT NULL CHECK (outcome IN ('YES', 'NO')),
    market_slug TEXT,
    side TEXT NOT NULL CHECK (side IN ('BUY', 'SELL')),
    requested_notional_pusd NUMERIC(18, 6) NOT NULL,
    requested_price_constraint NUMERIC(10, 6) NOT NULL,
    realized_avg_price NUMERIC(10, 6),
    realized_size_shares NUMERIC(18, 6),
    realized_notional_pusd NUMERIC(18, 6),
    fee_pusd NUMERIC(18, 6) NOT NULL DEFAULT 0,
    slippage_bps NUMERIC(12, 4),
    wallet_path TEXT,
    clob_order_id TEXT,
    trade_ids TEXT[] NOT NULL DEFAULT '{}',
    reject_reason TEXT,
    submitted_at TIMESTAMPTZ,
    posted_at TIMESTAMPTZ,
    filled_at TIMESTAMPTZ,
    cancelled_at TIMESTAMPTZ,
    rejected_at TIMESTAMPTZ,
    reconciled_at TIMESTAMPTZ,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    raw_execution_payload JSONB NOT NULL DEFAULT '{}'::jsonb,
    raw_fill_payloads JSONB NOT NULL DEFAULT '[]'::jsonb
);

CREATE INDEX idx_trade_journal_execution_id ON trade_journal (execution_id);
CREATE INDEX idx_trade_journal_proposal_id ON trade_journal (proposal_id);
CREATE INDEX idx_trade_journal_token_id ON trade_journal (token_id);
CREATE INDEX idx_trade_journal_state ON trade_journal (state);
CREATE INDEX idx_trade_journal_created_at ON trade_journal (created_at);
CREATE INDEX idx_trade_journal_mode ON trade_journal (mode);
```

### Deferred table: `wallet_registry`

`wallet_registry` is a Phase 3 table, not a Phase 0A deliverable. It should not be implemented in Phase 0A migrations, but downstream design should reserve the concept for wallet labels, cohort assignment, and historical edge measurements.
